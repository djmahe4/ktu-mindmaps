# ⚡ Fast Conditions Cheat Sheet

## 🔹 1. Single Loop

### ✅ Linear → **O(n)**

👉 Condition:

* `i++`, `i--`, `i += k`, `i -= k`

💡 Rule:

> Moves one step each time

**Example**

```c
for(i=1; i<=n; i++)
```

---

### ✅ Logarithmic → **O(log n)**

👉 Condition:

* `i *= c`, `i /= c` (c > 1)

💡 Rule:

> Value grows/shrinks exponentially

**Example**

```c
for(i=1; i<=n; i*=2)
```

---

## 🔹 2. Nested Loops

### ✅ Independent → **O(n²)**

👉 Condition:

* Inner loop does **not depend on outer**

💡 Rule:

> Full loop runs every time

**Example**

```c
for(i=1; i<=n; i++)
  for(j=1; j<=n; j++)
```

---

### ✅ Dependent → **O(n²)** (still)

👉 Condition:

* Inner loop depends on `i` (like `j <= i`)

💡 Rule:

> Forms triangle → sum = n²/2

**Example**

```c
for(i=1; i<=n; i++)
  for(j=1; j<=i; j++)
```

---

## 🔹 3. Mixed Pattern

### ✅ Linear + Log → **O(n log n)**

👉 Condition:

* One loop linear, inner loop logarithmic

**Example**

```c
for(i=1; i<=n; i++)
  for(j=1; j<=n; j*=2)
```

---

## 🔹 4. Quick Recognition Rules

* ➕ Add loops → **O(n + n) = O(n)**
* ✖️ Nested loops → **multiply → O(n × n) = O(n²)**
* 🔁 Halving/doubling → **log n**
* 🔺 Triangle loops → **O(n²)**

## Asymptotic notations 
These are mathematical tools used to describe the limiting behavior of functions, particularly in computer science, to analyze the efficiency of algorithms as the input size grows. They provide a standardized way to express how an algorithm's running time or space requirements scale with increasing input. This analysis focuses on the "order of growth" of an algorithm's basic operations, ignoring constant factors and lower-order terms that become negligible for large inputs.

Here are the five key asymptotic notations:

### 1. Big-Oh (O) Notation

**Definition:** Big-Oh notation describes the **upper bound** of an algorithm's running time, representing its **worst-case scenario**.

Mathematically, a function `f(n)` is said to be `O(g(n))` if there exist positive constants `c` and `n₀` such that `0 ≤ f(n) ≤ c * g(n)` for all `n ≥ n₀`.

**Graphical Interpretation:** Imagine a graph where the X-axis represents the input size `n` and the Y-axis represents time/effort. The function `f(n)` (the algorithm's runtime) will always lie **on or below** the curve `c * g(n)` for all `n` values greater than or equal to `n₀`. This means `c * g(n)` acts as an upper ceiling for `f(n)`.

```mermaid
graph TD
    A[f(n): Algorithm Runtime] --> B{Upper Bounded By};
    B --> C[c * g(n): Upper Bound];
    C --> D(For n >= n₀, f(n) <= c * g(n));
    style A fill:#f9f,stroke:#333,stroke-width:2px
    style C fill:#ccf,stroke:#333,stroke-width:2px
    style D fill:#afa,stroke:#333,stroke-width:2px
```

**Mnemonic:** Think of "Big-Oh" as "Oh no, it could be *at most* this bad!" – It gives the worst-case performance.

### 2. Big-Omega (Ω) Notation

**Definition:** Big-Omega notation describes the **lower bound** of an algorithm's running time, representing its **best-case scenario**.

Mathematically, a function `f(n)` is said to be `Ω(g(n))` if there exist positive constants `c` and `n₀` such that `0 ≤ c * g(n) ≤ f(n)` for all `n ≥ n₀`.

**Graphical Interpretation:** On a graph, `f(n)` (the algorithm's runtime) will always lie **on or above** the curve `c * g(n)` for all `n` values greater than or equal to `n₀`. This means `c * g(n)` acts as a floor for `f(n)`, guaranteeing a minimum level of performance.

```mermaid
graph TD
    A[f(n): Algorithm Runtime] --> B{Lower Bounded By};
    B --> C[c * g(n): Lower Bound];
    C --> D(For n >= n₀, f(n) >= c * g(n));
    style A fill:#f9f,stroke:#333,stroke-width:2px
    style C fill:#ccf,stroke:#333,stroke-width:2px
    style D fill:#afa,stroke:#333,stroke-width:2px```

**Mnemonic:** Think of "Big-Omega" as "Oh yeah, it will be *at least* this good!" – It gives the best-case performance.

### 3. Big-Theta (Θ) Notation

**Definition:** Big-Theta notation provides a **tight bound** for an algorithm's running time, meaning it's bounded both above and below by the same function (up to constant factors). This represents the **average-case scenario** or when the best and worst cases are asymptotically the same.

Mathematically, a function `f(n)` is said to be `Θ(g(n))` if there exist positive constants `c₁`, `c₂`, and `n₀` such that `0 ≤ c₁ * g(n) ≤ f(n) ≤ c₂ * g(n)` for all `n ≥ n₀`.

**Graphical Interpretation:** On a graph, `f(n)` (the algorithm's runtime) will be "sandwiched" between two curves, `c₁ * g(n)` and `c₂ * g(n)`, for all `n` values greater than or equal to `n₀`. This indicates that `f(n)` grows at the same rate as `g(n)`.

```mermaid
graph TD
    A[f(n): Algorithm Runtime] --> B{Bounded By};
    B --> C1[c1 * g(n): Lower Bound];
    B --> C2[c2 * g(n): Upper Bound];
    C1 --> D(For n >= n₀, c1*g(n) <= f(n) <= c2*g(n));
    C2 --> D;
    style A fill:#f9f,stroke:#333,stroke-width:2px
    style C1 fill:#ccf,stroke:#333,stroke-width:2px
    style C2 fill:#ccf,stroke:#333,stroke-width:2px
    style D fill:#afa,stroke:#333,stroke-width:2px
```

**Mnemonic:** Think of "Big-Theta" as "Tight fit!" – The algorithm's performance is tightly constrained between an upper and lower bound.

### 4. Little-oh (o) Notation

**Definition:** Little-oh notation describes a **strict upper bound** for an algorithm's running time. It indicates that a function `f(n)` grows *strictly slower* than `g(n)`. Unlike Big-Oh, it means `f(n)` becomes insignificant compared to `g(n)` as `n` approaches infinity.

Mathematically, a function `f(n)` is `o(g(n))` if for any positive constant `c > 0`, there exists a constant `n₀` such that `0 ≤ f(n) < c * g(n)` for all `n ≥ n₀`. Equivalently, `lim (f(n) / g(n)) = 0` as `n → ∞`.

**Graphical Interpretation:** On a graph, for sufficiently large `n`, `f(n)` will always be **below** `c * g(n)`, no matter how small `c` is (as long as it's positive). This implies `g(n)` grows *much faster* than `f(n)`.

```mermaid
graph TD
    A[f(n): Algorithm Runtime] --> B{Strictly Upper Bounded By};
    B --> C[c * g(n): Strict Upper Bound];
    C --> D(For n >= n₀, f(n) < c * g(n) AND lim(f(n)/g(n)) = 0);
    style A fill:#f9f,stroke:#333,stroke-width:2px
    style C fill:#ccf,stroke:#333,stroke-width:2px
    style D fill:#afa,stroke:#333,stroke-width:2px
```

**Mnemonic:** Think of "Little-oh" as "Oh, so small!" – `f(n)` is negligible compared to `g(n)`.

### 5. Little-omega (ω) Notation

**Definition:** Little-omega notation describes a **strict lower bound**. It indicates that a function `f(n)` grows *strictly faster* than `g(n)`. It is the inverse of little-oh notation.

Mathematically, a function `f(n)` is `ω(g(n))` if for any positive constant `c > 0`, there exists a constant `n₀` such that `f(n) > c * g(n) ≥ 0` for all `n ≥ n₀`. Equivalently, `lim (f(n) / g(n)) = ∞` as `n → ∞`.

**Graphical Interpretation:** On a graph, for sufficiently large `n`, `f(n)` will always be **above** `c * g(n)`, no matter how large `c` is. This implies `f(n)` grows *much faster* than `g(n)`.

```mermaid
graph TD
    A[f(n): Algorithm Runtime] --> B{Strictly Lower Bounded By};
    B --> C[c * g(n): Strict Lower Bound];
    C --> D(For n >= n₀, f(n) > c * g(n) AND lim(f(n)/g(n)) = ∞);
    style A fill:#f9f,stroke:#333,stroke-width:2px
    style C fill:#ccf,stroke:#333,stroke-width:2px
    style D fill:#afa,stroke:#333,stroke-width:2px
```

**Mnemonic:** Think of "Little-omega" as "Way bigger!" – `f(n)` significantly outgrows `g(n)`.

### Properties of Asymptotic Notations

Asymptotic notations possess several properties that are useful in algorithm analysis:

*   **Reflexivity:** A function is in the same asymptotic class as itself.
    *   `f(n) = O(f(n))`
    *   `f(n) = Ω(f(n))`
    *   `f(n) = Θ(f(n))`
*   **Symmetry:** If `f(n)` is tightly bound by `g(n)`, then `g(n)` is also tightly bound by `f(n)`.
    *   `f(n) = Θ(g(n))` if and only if `g(n) = Θ(f(n))`
*   **Transpose Symmetry:** Relates upper and lower bounds.
    *   `f(n) = O(g(n))` if and only if `g(n) = Ω(f(n))`
    *   `f(n) = o(g(n))` if and only if `g(n) = ω(f(n))`
*   **Transitivity:** If `f(n)` is bounded by `g(n)`, and `g(n)` is bounded by `h(n)`, then `f(n)` is also bounded by `h(n)`.
    *   If `f(n) = O(g(n))` and `g(n) = O(h(n))`, then `f(n) = O(h(n))`
    *   If `f(n) = Ω(g(n))` and `g(n) = Ω(h(n))`, then `f(n) = Ω(h(n))`
    *   If `f(n) = Θ(g(n))` and `g(n) = Θ(h(n))`, then `f(n) = Θ(h(n))`
    *   If `f(n) = o(g(n))` and `g(n) = o(h(n))`, then `f(n) = o(h(n))`
    *   If `f(n) = ω(g(n))` and `g(n) = ω(h(n))`, then `f(n) = ω(h(n))`

These notations are crucial for comparing algorithms and predicting their performance for large inputs, abstracting away machine-dependent factors and implementation details.
