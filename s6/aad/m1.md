# ⚡ Fast Conditions Cheat Sheet

## 🔹 1. Single Loop

### ✅ Linear → **O(n)**

👉 Condition:

* `i++`, `i--`, `i += k`, `i -= k`

💡 Rule:

> Moves one step each time

**Example**

```c
for(i=1; i<=n; i++)
```

---

### ✅ Logarithmic → **O(log n)**

👉 Condition:

* `i *= c`, `i /= c` (c > 1)

💡 Rule:

> Value grows/shrinks exponentially

**Example**

```c
for(i=1; i<=n; i*=2)
```

---

### ✅ Square Root → **O(√n)**

👉 Condition:

* `i * i <= n` or `i <= √n`

💡 Rule:

> Loop runs until the square of the index reaches n

**Example**

```c
for(i=1; i*i<=n; i++)
```

---

## 🔹 2. Nested Loops

### ✅ Independent → **O(n²)**

👉 Condition:

* Inner loop does **not depend on outer**

💡 Rule:

> Full loop runs every time

**Example**

```c
for(i=1; i<=n; i++)
  for(j=1; j<=n; j++)
```

---

### ✅ Dependent → **O(n²)** (still)

👉 Condition:

* Inner loop depends on `i` (like `j <= i`)

💡 Rule:

> Forms triangle → sum = n²/2

**Example**

```c
for(i=1; i<=n; i++)
  for(j=1; j<=i; j++)
```

---

## 🔹 3. Mixed Pattern

### ✅ Linear + Log → **O(n log n)**

👉 Condition:

* One loop linear, inner loop logarithmic

**Example**

```c
for(i=1; i<=n; i++)
  for(j=1; j<=n; j*=2)
```

---

## 🔹 4. Quick Recognition Rules

* ➕ Add loops → **O(n + n) = O(n)**
* ✖️ Nested loops → **multiply → O(n × n) = O(n²)**
* 🔁 Halving/doubling → **log n**
* 🔺 Triangle loops → **O(n²)**
* 🔢 Index squared check → **√n**

### Properties of Asymptotic Notations

Asymptotic notations possess several properties that are useful in algorithm analysis:

*   **Reflexivity:** A function is in the same asymptotic class as itself.
    *   `f(n) = O(f(n))`
    *   `f(n) = Ω(f(n))`
    *   `f(n) = Θ(f(n))`
*   **Symmetry:** If `f(n)` is tightly bound by `g(n)`, then `g(n)` is also tightly bound by `f(n)`.
    *   `f(n) = Θ(g(n))` if and only if `g(n) = Θ(f(n))`
*   **Transpose Symmetry:** Relates upper and lower bounds.
    *   `f(n) = O(g(n))` if and only if `g(n) = Ω(f(n))`
    *   `f(n) = o(g(n))` if and only if `g(n) = ω(f(n))`
*   **Transitivity:** If `f(n)` is bounded by `g(n)`, and `g(n)` is bounded by `h(n)`, then `f(n)` is also bounded by `h(n)`.
    *   If `f(n) = O(g(n))` and `g(n) = O(h(n))`, then `f(n) = O(h(n))`
    *   If `f(n) = Ω(g(n))` and `g(n) = Ω(h(n))`, then `f(n) = Ω(h(n))`
    *   If `f(n) = Θ(g(n))` and `g(n) = Θ(h(n))`, then `f(n) = Θ(h(n))`
    *   If `f(n) = o(g(n))` and `g(n) = o(h(n))`, then `f(n) = o(h(n))`
    *   If `f(n) = ω(g(n))` and `g(n) = ω(h(n))`, then `f(n) = ω(h(n))`

These notations are crucial for comparing algorithms and predicting their performance for large inputs, abstracting away machine-dependent factors and implementation details.

## 1. Linear Search: Detailed Complexity Analysis

### Algorithm Description
Linear search, also known as sequential search, is the simplest searching algorithm. It works by iterating through each element of a list or array, one by one, and comparing it with the target value. The search continues until the target element is found, or the entire list has been traversed without finding the element. It does not require the input list to be sorted.

### Pseudocode Segment

```pseudocode
FUNCTION linear_search(arr, target):
    n = LENGTH(arr)
    comparisons = 0

    FOR i FROM 0 TO n-1:
        comparisons = comparisons + 1  // Count comparison: arr[i] == target
        IF arr[i] == target:
            RETURN i, comparisons      // Target found at index i
    
    RETURN -1, comparisons             // Target not found
END FUNCTION
```

### Best Case Complexity: O(1) or Ω(1)

*   **Scenario:** The best case for linear search occurs when the target element is found at the very **first position** of the list.
*   **Explanation:** In this ideal scenario, the `FOR` loop in the pseudocode executes only once for `i = 0`. The `IF arr[i] == target` condition is met immediately. The search terminates after a single comparison.
*   **Walk-through Example:**
    *   `arr = [10, 20, 30, 40, 50]`
    *   `target = 10`
    1.  `i` is initialized to `0`. `comparisons` becomes `1`.
    2.  `arr[0]` (which is `10`) is compared with `target` (which is `10`).
    3.  They match. The function `RETURN`s `0` and `1`.
*   **Number of Operations (Comparisons):** 1
*   **Complexity Notation:** **O(1)** or **Ω(1)**. It's constant time, irrespective of the list's size `n`.

### Worst Case Complexity: O(n)

*   **Scenario:** The worst case for linear search happens in two situations:
    1.  The target element is located at the **last position** of the list.
    2.  The target element is **not present** anywhere in the list.
*   **Explanation:** In both scenarios, the `FOR` loop in the pseudocode must iterate through all `n` elements. For each `i` from `0` to `n-1`, a comparison is made. If the target is at the last position, the last comparison finds it. If not present, all `n` comparisons are made before the loop finishes and `-1` is returned.
*   **Walk-through Example (Target at last position):**
    *   `arr = [10, 20, 30, 40, 50]`
    *   `target = 50`
    1.  The loop runs for `i = 0, 1, 2, 3`. For each, `arr[i] != target`. `comparisons` increments `4` times.
    2.  For `i = 4`, `comparisons` becomes `5`. `arr[4]` (which is `50`) is compared with `target` (`50`).
    3.  They match. The function `RETURN`s `4` and `5`.
*   **Walk-through Example (Target not found):**
    *   `arr = [10, 20, 30, 40, 50]`
    *   `target = 60`
    1.  The `FOR` loop iterates from `i = 0` to `n-1` (i.e., `4`).
    2.  For each `i`, `arr[i]` is compared with `60`. All `5` comparisons result in `arr[i] != target`.
    3.  The loop completes. The function `RETURN`s `-1` and `5`.
*   **Number of Operations (Comparisons):** `n`
*   **Complexity Notation:** **O(n)**. The number of operations grows linearly with the input size `n`.

### Average Case Complexity: Θ(n)

*   **Scenario:** The target element is found somewhere in the **middle** of the list.
*   **Explanation:** Assuming the target element is present in the list and is equally likely to be at any position from `1` to `n`, the `FOR` loop will, on average, execute `(n+1)/2` times. This means approximately `n/2` comparisons are made.
*   **Walk-through Example:**
    *   `arr = [10, 20, 30, 40, 50]`
    *   `target = 30`
    1.  `i = 0`, `arr[0]` (10) != 30. `comparisons = 1`.
    2.  `i = 1`, `arr[1]` (20) != 30. `comparisons = 2`.
    3.  `i = 2`, `arr[2]` (30) == 30. `comparisons = 3`. Match found.
    4.  The function `RETURN`s `2` and `3`.
*   **Number of Operations (Comparisons):** Approximately `n/2`
*   **Complexity Notation:** **Θ(n)**. The average number of operations still scales linearly with `n`.

## 2. Binary Search: Detailed Complexity Analysis

### Algorithm Description
Binary search is a highly efficient algorithm for finding an element in a **sorted** list or array. It works by repeatedly dividing the search interval in half. It compares the target value with the middle element of the interval. If they match, the search is successful. If the target is smaller, the search continues in the left half; if larger, in the right half. This process is repeated until the element is found or the interval becomes empty.

### Pseudocode Segment

```pseudocode
FUNCTION binary_search(arr, target):
    low = 0
    high = LENGTH(arr) - 1
    comparisons = 0

    WHILE low <= high:
        mid = (low + high) // 2
        comparisons = comparisons + 1  // Count comparison: arr[mid] == target

        IF arr[mid] == target:
            RETURN mid, comparisons
        ELSE IF arr[mid] < target:
            low = mid + 1
        ELSE: // arr[mid] > target
            high = mid - 1
    
    RETURN -1, comparisons             // Target not found
END FUNCTION
```

### Best Case Complexity: O(1) or Ω(1)

*   **Scenario:** The best case occurs when the target element is the **middle element** of the initial search interval.
*   **Explanation:** In this ideal scenario, the `WHILE` loop in the pseudocode executes only once. The `arr[mid] == target` condition is met in the first iteration. The search terminates after a single comparison.
*   **Walk-through Example:**
    *   `arr = [10, 20, 30, 40, 50]` (n=5)
    *   `target = 30`
    1.  `low = 0`, `high = 4`.
    2.  `mid = (0 + 4) // 2 = 2`. `comparisons` becomes `1`.
    3.  `arr[2]` (which is `30`) is compared with `target` (which is `30`).
    4.  They match. The function `RETURN`s `2` and `1`.
*   **Number of Operations (Comparisons):** 1
*   **Complexity Notation:** **O(1)** or **Ω(1)**.

### Worst Case Complexity: O(log n)

*   **Scenario:** The worst case for binary search occurs when the target element is located at either **end** of the list (e.g., first or last element), or when the target element is **not present** in the list.
*   **Explanation:** In these situations, the `WHILE` loop continues to divide the search space by half in each iteration until `low` becomes greater than `high`. Each iteration performs a comparison and potentially updates `low` or `high`. The number of times you can divide `n` by 2 until it reaches 1 is `log₂n`.
*   **Walk-through Example (Target at end):**
    *   `arr = [10, 20, 30, 40, 50]` (n=5)
    *   `target = 50`
    1.  `low = 0`, `high = 4`. `mid = 2`. `arr[2]` (30) < 50. `low = 3`. `comparisons = 1`.
    2.  `low = 3`, `high = 4`. `mid = 3`. `arr[3]` (40) < 50. `low = 4`. `comparisons = 2`.
    3.  `low = 4`, `high = 4`. `mid = 4`. `arr[4]` (50) == 50. `comparisons = 3`. Match found.
    4.  The function `RETURN`s `4` and `3`.
*   **Walk-through Example (Target not found):**
    *   `arr = [10, 20, 30, 40, 50]` (n=5)
    *   `target = 25`
    1.  `low = 0`, `high = 4`. `mid = 2`. `arr[2]` (30) > 25. `high = 1`. `comparisons = 1`.
    2.  `low = 0`, `high = 1`. `mid = 0`. `arr[0]` (10) < 25. `low = 1`. `comparisons = 2`.
    3.  `low = 1`, `high = 1`. `mid = 1`. `arr[1]` (20) < 25. `low = 2`. `comparisons = 3`.
    4.  `low = 2`, `high = 1`. `low` > `high`. Loop terminates.
    5.  The function `RETURN`s `-1` and `3`.
*   **Number of Operations (Comparisons):** Approximately `log₂n`
*   **Complexity Notation:** **O(log n)**.

### Average Case Complexity: Θ(log n)

*   **Scenario:** The target element is found after a few divisions, not necessarily in the absolute middle.
*   **Explanation:** On average, binary search still performs a logarithmic number of comparisons. The depth of the node where an element is found contributes to the number of comparisons. Summing the depths of all possible nodes and dividing by `n` yields a value that is still proportional to `log n`.
*   **Number of Operations (Comparisons):** Approximately `log₂n`
*   **Complexity Notation:** **Θ(log n)**.

## 3. Insertion Sort: Detailed Complexity Analysis

### Algorithm Description
Insertion sort builds the final sorted array one item at a time. It iterates through the input list, and for each element, it finds the correct position within the already sorted part of the list and inserts it there. This involves shifting elements to the right to make space for the new element.

### Pseudocode Segment

```pseudocode
FUNCTION insertion_sort(arr):
    n = LENGTH(arr)
    operations = 0 // Tracks comparisons and shifts

    FOR i FROM 1 TO n-1: // Iterate from the second element
        key = arr[i]     // Element to be inserted
        j = i - 1        // Index of the last element in the sorted part

        // Move elements of arr[0..i-1], that are greater than key,
        // to one position ahead of their current position
        WHILE j >= 0 AND arr[j] > key:
            operations = operations + 1 // Count comparison: arr[j] > key
            arr[j + 1] = arr[j]         // Shift operation
            operations = operations + 1 // Count shift
            j = j - 1
        
        // After WHILE loop, one final comparison arr[j] > key occurred
        // that resulted in false (either j < 0 or arr[j] <= key).
        // If j >= 0, this comparison should be counted.
        IF j >= 0:
            operations = operations + 1 // The comparison that made WHILE condition false

        arr[j + 1] = key            // Insert the key element
        operations = operations + 1 // Count insertion
        
    RETURN arr, operations
END FUNCTION
```
*Note: The `operations` counter is designed to capture both comparisons and significant assignments/shifts for a more comprehensive (though still simplified) measure of work.*

### Best Case Complexity: O(n) or Ω(n)

*   **Scenario:** The best case for insertion sort occurs when the input array is **already sorted** in ascending order.
*   **Explanation:** In this scenario, for each element `arr[i]` (starting from `i=1`), the inner `WHILE` loop condition `arr[j] > key` will immediately evaluate to `FALSE` (because `arr[j]` is always less than or equal to `key`). No elements need to be shifted. The `WHILE` loop makes only one comparison per `i` before `arr[j+1] = key` is executed.
*   **Walk-through Example:**
    *   `arr = [1, 2, 3, 4, 5]` (n=5)
    1.  `i = 1`, `key = 2`, `j = 0`.
        *   `WHILE` condition `j >= 0 AND arr[0] > key` (i.e., `0 >= 0 AND 1 > 2`) is `FALSE`. **(1 comparison)**
        *   `arr[1] = key` (2). **(1 insertion)**
    2.  `i = 2`, `key = 3`, `j = 1`.
        *   `WHILE` condition `j >= 0 AND arr[1] > key` (i.e., `1 >= 0 AND 2 > 3`) is `FALSE`. **(1 comparison)**
        *   `arr[2] = key` (3). **(1 insertion)**
    3.  This continues for all `i`.
*   **Number of Operations:**
    *   `FOR` loop runs `n-1` times.
    *   Each `FOR` loop iteration involves 1 comparison and 1 insertion.
    *   Total operations: Approximately `2(n-1)`.
*   **Complexity Notation:** **O(n)** or **Ω(n)**.

### Worst Case Complexity: O(n²)

*   **Scenario:** The worst case for insertion sort occurs when the input array is **sorted in reverse order** (descending).
*   **Explanation:** In this situation, for each element `arr[i]`, the inner `WHILE` loop condition `arr[j] > key` will always be `TRUE` for all `j` from `i-1` down to `0`. This means `arr[i]` must be compared with all `i-1` elements in the sorted portion, and all `i-1` elements must be shifted one position to the right.
*   **Walk-through Example:**
    *   `arr = [5, 4, 3, 2, 1]` (n=5)
    1.  `i = 1`, `key = 4`, `j = 0`.
        *   `WHILE` `0 >= 0 AND arr[0] (5) > key (4)` is `TRUE`. **(1 comparison)**
        *   `arr[1] = arr[0]` (5). `j = -1`. **(1 shift)**
        *   `WHILE` condition `j >= 0` is `FALSE`.
        *   `arr[0] = key` (4). **(1 insertion)**
        *   Current `arr = [4, 5, 3, 2, 1]`
    2.  `i = 2`, `key = 3`, `j = 1`.
        *   `WHILE` `1 >= 0 AND arr[1] (5) > key (3)` is `TRUE`. **(1 comparison)**
        *   `arr[2] = arr[1]` (5). `j = 0`. **(1 shift)**
        *   `WHILE` `0 >= 0 AND arr[0] (4) > key (3)` is `TRUE`. **(1 comparison)**
        *   `arr[1] = arr[0]` (4). `j = -1`. **(1 shift)**
        *   `arr[0] = key` (3). **(1 insertion)**
        *   Current `arr = [3, 4, 5, 2, 1]`
    3.  This pattern continues, with approximately `i` comparisons and `i` shifts/insertions for each `i`.
*   **Number of Operations:**
    *   The `FOR` loop runs `n-1` times.
    *   The inner `WHILE` loop performs `i` operations (approx) for each `i`.
    *   Total operations: Approximately `1 + 2 + ... + (n-1) = n(n-1)/2`.
    *   The number of comparisons and shifts/assignments are each roughly `n²/2`.
    *   Overall: The total number of operations is proportional to `n²`.
*   **Complexity Notation:** **O(n²)**.

### Average Case Complexity: Θ(n²)

*   **Scenario:** A randomly ordered array.
*   **Explanation:** On average, for each element `arr[i]`, it will need to be compared with roughly half of the elements in the sorted subarray before finding its correct position. This will also involve shifting, on average, half of those elements. The total number of operations (comparisons and shifts combined) will still be proportional to `n²`. The sum `(1 + 2 + ... + (n-1))` is divided by some constant factor, but the order of growth remains `n²`.
*   **Number of Operations:** Approximately `n²/4` comparisons and `n²/4` shifts.
*   **Complexity Notation:** **Θ(n²)**.
### Summary

* Insertion Sort: best O(n), worst O(n²), average O(n²)
* Linear Search: best O(1), worst O(n), average O(n/2)
* Binary Search: best O(1), worst O(log n)
