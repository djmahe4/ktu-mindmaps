Data Flow Model (DFD)

Shows how data moves through processes, stores, and external entities.

When to use

  • Early analysis to clarify what the system does with data.
  • Communicating with non-technical stakeholders.

Key points to remember

  • Processes transform inputs → outputs.
  • Data stores hold information between processes.
  • External entities are sources/sinks outside the system.
  • Levels: Context (Level-0) → Level-1/2 decomposition.
Notation: Circle/rounded-rect = Process, Open-ended rect = Data store, Square = External entity, Arrow = Data flow.

Entity–Relationship (ER) Model

Conceptual view of data: entities, attributes, and relationships.

When to use

  • Designing databases before logical/physical schemas.
  • Clarifying business data and rules with stakeholders.

Key points to remember

  • Entity = thing of interest; Attribute = property; Relationship = association.
  • Cardinality: 1–1, 1–N, M–N; Participation: total/partial.
  • Keys: primary key uniquely identifies an entity instance.
  • Weak entities depend on owners (double border).
Notation: Rectangle = Entity, Ellipse = Attribute, Diamond = Relationship.

Class Diagram (UML)

Static structure of classes, attributes, operations, and relationships.

When to use

  • Designing OO systems and APIs.
  • Refactoring legacy code into clear abstractions.

Key points to remember

  • Relationships: association, aggregation (hollow diamond), composition (filled diamond), inheritance (triangle).
  • Visibility: + public, - private, # protected.
  • Favor high cohesion, low coupling; apply SOLID principles.

State Machine Model

Dynamic behavior: states and transitions triggered by events/conditions.

When to use

  • Reactive systems, device controllers, workflows.
  • When behavior depends on history (current state).

Key points to remember

  • Elements: state, event, guard [condition], action / activity.
  • Initial and final pseudo-states; entry/exit actions.
  • Consider hierarchical and concurrent states for complexity.

Component Model

High-level organization of deployable units with provided/required interfaces.

When to use

  • Service decomposition, microservices, plug‑in architectures.
  • Defining clear contracts between parts of a system.

Key points to remember

  • Component = replaceable unit with interfaces.
  • Ports: provided (what it offers) vs required (what it needs).
  • Use explicit interfaces to reduce coupling and increase reuse.

Deployment Model

Physical view: nodes (hardware/VMs/containers), communication links, and deployed artifacts.

When to use

  • Capacity planning, network/security design, DevOps handover.
  • Identifying single points of failure and latency hotspots.

Key points to remember

  • Nodes host artifacts (services, DBs, UIs).
  • Links show protocols (HTTP, gRPC) and QoS (latency, bandwidth).
  • Consider redundancy, scaling strategy, and zoning.
